## 📌 TL;DR（执行摘要）

- **核心目标**：开发一款仿《剑与远征》的东方修仙放置 RPG MVP，使用 Godot 4 引擎，验证"闭关修炼→斗法推关→寻仙抽卡→突破境界"核心循环的乐趣性
- **关键决策**：Godot 4 + GDScript / 东方修仙题材 / 移动端优先（微信小游戏首发）/ 考虑商业化（F2P+激励广告+轻度IAP）/ 1人独立开发+AI辅助美术
- **MVP 范围**：P0 共 9 个需求（34 人天），覆盖 6 大核心系统 + 3 个支撑系统；P1 共 4 个需求（13 人天），上线后 2 周内迭代
- **时间线**：中性估算 21 周（约 5 个月），乐观 15 周，保守 28.5 周
- **下一步**：确认 PRD 待确认问题（美术风格/品质阶数/修士数量/开发周期），启动 Phase 0 入门筑基

| # | 目标 | 维度 | 衡量指标 | 目标值 |
|---|------|------|----------|--------|
| G1 | **验证核心修仙放置循环的乐趣性** — 玩家觉得"挂机修炼→收菜→推关→突破"的循环有趣且上瘾 | 玩家体验 | D1 留存率 / D7 留存率 / 平均单次会话时长 | D1 ≥ 25%，D7 ≥ 7%，单次会话 5-8 分钟 |

## 2. 用户故事

### US-1：上班族修仙者
> 作为一名朝九晚五的上班族修仙者，我希望在上班期间我的角色能自动修炼积累灵石和经验，以便我午休时打开手机收菜，花 5 分钟推几关、突破一下境界，获得成长的满足感。

### US-2：碎片时间修仙者
> 作为一名只有通勤和睡前碎片时间的修仙者，我希望每次打开游戏 3-5 分钟就能完成核心操作（收菜→推关→升级→抽卡），以便我不会因为没时间而落后，也不会因为需要长时间在线而感到负担。

### US-3：初入仙门的新弟子
> 作为一名第一次接触放置 RPG 的新弟子，我希望游戏在前 10 分钟教会我修炼（挂机）、突破（升级）、寻仙（抽卡）的核心玩法，以便我不会因为系统太多而迷茫放弃。

### US-4：策略型修仙者
> 作为一名喜欢研究阵容搭配的策略型修仙者，我希望阵营克制和站位策略能让我以弱胜强，以便我在遇到渡劫瓶颈时，可以通过调整阵容和策略来突破，而不是只能靠纯数值碾压。

### US-5：愿意支持的修仙者
> 作为一名愿意为好游戏付费的修仙者，我希望付费能让我获得合理的便利和外观奖励，而不是直接买到碾压免费玩家的数值，以便我既能支持开发者，又不会觉得游戏变成了"充钱就能赢"。

## 3. 用户研究洞察（来自瑞思）

### 3.1 玩家最大痛点

| 痛点 | 说明 | 对本项目的启示 |
|------|------|----------------|
| 数值膨胀过快 | AFK 后期数值天文数字，玩家失去感知 | 属性曲线设计要平滑，修仙境界体系天然分层，每境界内数值跨度可控 |
| P2W（充钱就能赢） | 付费玩家与免费玩家差距过大 | MVP 阶段不做 PvP，付费以便利性+外观为主，限时礼包提供加速而非碾压 |
| 系统过载 | AFK 后期 60+ 系统，玩家疲劳 | **MVP 系统数控制在 10 个以内**，核心循环清晰 |

### 3.2 玩家核心需求（P0）

| 需求 | 优先级 | 本项目对应 |
|------|--------|-----------|
| 碎片化时间利用 | P0 | 离线收益 + 5 分钟会话设计 |
| 成长反馈与成就感 | P0 | 密集奖励曲线 + 突破动画 + 境界晋升仪式感 |
| 公平的竞争环境 | P1 | MVP 无 PvP，后续论剑系统做匹配制 |

### 3.3 新手体验建议

- **精简系统至 10 个以内**：MVP 只做 6 大核心系统 + 3 个支撑系统 = 9 个
- **设计"安全网"机制**：前期隐藏降难 Buff，玩家感觉自己在"变强"而非游戏在"放水"
- **首周密集奖励曲线**：前 60 分钟领取 70 次奖励，首日大量灵石/抽卡券，制造"上瘾"窗口

### 3.4 独立游戏用户画像

- 40% 过去一年买过独立游戏
- 2D 风格统一即可（不追求 3A 画质）
- 音乐是差异化机会 → 修仙题材配国风 BGM 是天然优势

## 7. 关键流程图 / UI 设计描述

### 7.1 核心玩法循环

```
闭关修炼(挂机产出) → 斗法(推关) → 渡劫瓶颈(卡关)
     ↑                                      ↓
  修炼升级(升级) ←── 调整阵容(策略) ←── 突破境界(升品质) ←── 寻仙(抽卡)
```

### 7.2 核心用户流程

**流程 1：每日登录 → 收菜 → 推关**
启动游戏 → 加载存档 → 计算离线收益 → 主界面（"闭关结束"弹窗）→ 领取 → 灵石/修为入账 → 点击"境界"入口 → 关卡地图 → 选择关卡 → 斗法 → 自动战斗+手动绝学 → 胜利 → 结算奖励 → 继续推关/卡关返回升级

**流程 2：卡关 → 升级 → 突破**
渡劫失败 → 返回主界面 → 提示"修为不足" → 修士界面 → 消耗修为+灵尘升级 → 有同名碎片则突破境界 → 突破成功+动画 → 解锁新技能 → 返回关卡重新挑战

**流程 3：攒够灵石 → 寻仙**
灵石充足 → 寻仙界面 → 单抽/十连 → 抽卡动画 → 展示结果 → 重复修士转碎片 → 新修士解锁图鉴 → 调整阵容

### 9.3 数据驱动设计（JSON 配置表）

**核心原则：所有数值平衡通过 JSON 配置，不改代码即可调参。**

**balance.json 关键配置：**
```json
{
  "idle": {"max_offline_hours": 12, "spirit_stone_rate_per_hour": 100, "exp_rate_per_hour": 800,
           "offline_coefficient": {"spirit_stones": 0.6, "exp": 0.9}},
  "faction_counter": {"advantage_multiplier": 1.25, "disadvantage_multiplier": 0.75,
                      "same_faction_bonus": {"attack": 1.25, "hp": 1.25, "required_count": 5}},
  "gacha": {"single_cost": 200, "ten_pull_cost": 2000, "pity_threshold": 30,
            "rates": {"blue": 0.75, "purple": 0.20, "gold": 0.045, "red": 0.005}},
  "early_game_safety_net": {"stages_1_to_20_enemy_debuff": 0.15, "hidden": true}
}
```

**属性计算公式：**
```
属性 = (基础 + 等级 × 成长) × 品质系数 + 品质固定值
```

### 9.4 GDScript 开发要点（给 Java/JS 开发者）

| Java/JS 概念 | GDScript 对应 | 说明 |
|-------------|---------------|------|
| Class | `class_name` + `extends` | 每个文件默认一个类 |
| Event/EventEmitter | **Signal** | `signal level_up` / `level_up.emit()` |
| addEventListener | `signal.connect(callable)` | 事件驱动解耦 |
| Constructor | `_ready()` | 节点进入场景树时调用 |
| requestAnimationFrame | `_process(delta)` | 每帧调用 |
| Promise/async-await | `await` + Coroutine | 原生协程 |
| public field | `@export` | 暴露到编辑器 |
| Singleton | Autoload 单例 | Project Settings 注册 |

**关键建议：**
1. 强烈建议使用类型标注（对 Java/JS 开发者更友好，且有性能优势）
2. 系统间通信用 Signal 解耦，不要直接互相调用
3. 节点树类似 DOM，用路径访问子节点 `$VBoxContainer/HPBar`
4. JSON 读取用 `FileAccess.open()` + `JSON.new().parse()`

### 9.5 离线收益实现方案

```gdscript
func calculate_offline_rewards(last_timestamp: int) -> Dictionary:
    var now = Time.get_unix_time_from_system()
    var elapsed_hours = min((now - last_timestamp) / 3600.0, 12.0)
    if elapsed_hours < 1.0 / 60.0: return {}
    var balance = load_json("res://data/balance.json").idle
    return {
        "spirit_stones": int(balance.spirit_stone_rate_per_hour * elapsed_hours * 0.6),
        "exp": int(balance.exp_rate_per_hour * elapsed_hours * 0.9),
        "offline_hours": elapsed_hours
    }
```

### 9.6 移动端适配

| 问题 | 方案 |
|------|------|
| **渲染器** | **必须选 Compatibility**（OpenGL ES 3.0），避免 Vulkan 兼容问题 |
| **屏幕适配** | 基准 1080×1920，`stretch_mode = canvas_items`，`aspect = keep` |
| **安全区域** | `DisplayServer.get_display_safe_area()` |
| **触控输入** | `Control.gui_input`，按钮 ≥ 48×48 dp |
| **内存管理** | 修士立绘按需 `load()`，切换场景 `queue_free()`，纹理压缩 ASTC |
| **包体控制** | 目标 ≤ 50MB（微信小游戏限制） |
| **后台/前台切换** | 监听 `will_focus_out`，退出时存档，前台时计算离线收益 |
