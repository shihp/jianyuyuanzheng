## 1. 产品目标

三个目标相互正交，分别覆盖**玩家体验**、**开发效率**和**商业验证**三个维度：

| # | 目标 | 维度 | 衡量指标 | 目标值 |
|---|------|------|----------|--------|
| G1 | **验证核心修仙放置循环的乐趣性** — 玩家觉得"挂机修炼→收菜→推关→突破"的循环有趣且上瘾 | 玩家体验 | D1 留存率 / D7 留存率 / 平均单次会话时长 | D1 ≥ 25%，D7 ≥ 7%，单次会话 5-8 分钟 |
| G2 | **建立可持续的内容生产管线** — 1 人可通过数据配置在 1 天内新增 1 个修士或 1 章关卡，无需改代码 | 开发效率 | 新增英雄平均耗时 / 新增章节平均耗时 | 英雄 ≤ 0.5 天，章节 ≤ 1 天 |
| G3 | **验证修仙题材付费意愿** — 玩家愿意为修仙放置 RPG 付费，且付费不破坏免费玩家体验 | 商业验证 | 付费率 / 首周 IAP 转化率 / ARPU | 付费率 ≥ 2%，首周 IAP 转化 ≥ 1.5% |

## 2. 用户故事

### US-1：上班族修仙者
> **作为一名**朝九晚五的上班族修仙者，
> **我希望**在上班期间我的角色能自动修炼积累灵石和经验，
> **以便**我午休时打开手机收菜，花 5 分钟推几关、突破一下境界，获得成长的满足感。

### US-2：碎片时间修仙者
> **作为一名**只有通勤和睡前碎片时间的修仙者，
> **我希望**每次打开游戏 3-5 分钟就能完成核心操作（收菜→推关→升级→抽卡），
> **以便**我不会因为没时间而落后，也不会因为需要长时间在线而感到负担。

### US-3：初入仙门的新弟子
> **作为一名**第一次接触放置 RPG 的新弟子，
> **我希望**游戏在前 10 分钟教会我修炼（挂机）、突破（升级）、寻仙（抽卡）的核心玩法，
> **以便**我不会因为系统太多而迷茫放弃。

### US-4：策略型修仙者
> **作为一名**喜欢研究阵容搭配的策略型修仙者，
> **我希望**阵营克制和站位策略能让我以弱胜强，
> **以便**我在遇到渡劫瓶颈时，可以通过调整阵容和策略来突破，而不是只能靠纯数值碾压。

### US-5：愿意支持的修仙者
> **作为一名**愿意为好游戏付费的修仙者，
> **我希望**付费能让我获得合理的便利和外观奖励，而不是直接买到碾压免费玩家的数值，
> **以便**我既能支持开发者，又不会觉得游戏变成了"充钱就能赢"。

## 3. 用户研究洞察

> 来源：瑞思（用户研究员）的前序调研报告

### 3.1 玩家最大痛点

| 痛点 | 说明 | 对本项目的启示 |
|------|------|----------------|
| 数值膨胀过快 | 后期数值天文数字，玩家失去感知 | 属性曲线设计要平滑，修仙境界体系天然分层（练气→筑基→...），每境界内数值跨度可控 |
| P2W（充钱就能赢） | 付费玩家与免费玩家差距过大 | MVP 阶段不做 PvP，付费以便利性+外观为主，限时礼包提供加速而非碾压 |
| 系统过载 | AFK 后期 60+ 系统，玩家疲劳 | **MVP 系统数控制在 10 个以内**，核心循环清晰 |

### 3.2 玩家核心需求（P0）

| 需求 | 优先级 | 本项目对应 |
|------|--------|-----------|
| 碎片化时间利用 | P0 | 离线收益 + 5 分钟会话设计 |
| 成长反馈与成就感 | P0 | 密集奖励曲线 + 突破动画 + 境界晋升仪式感 |
| 公平的竞争环境 | P1 | MVP 无 PvP，后续论剑系统做匹配制 |

### 3.3 新手体验建议

- **精简系统至 10 个以内**：MVP 只做 6 大核心系统 + 3 个支撑系统 = 9 个
- **设计"安全网"机制**：前期隐藏降难 Buff，玩家感觉自己在"变强"而非游戏在"放水"
- **首周密集奖励曲线**：前 60 分钟领取 70 次奖励，首日大量灵石/抽卡券，制造"上瘾"窗口

### 3.4 独立游戏用户画像

- 40% 过去一年买过独立游戏
- 2D 风格统一即可（不追求 3A 画质）
- 音乐是差异化机会 → 修仙题材配国风 BGM 是天然优势
- 买断制 $10-15 更受欢迎，但本产品走 F2P+内购路线（放置 RPG 品类惯例）

## 4. 竞品对比

### 4.1 AFK Arena 核心机制拆解

| 系统 | AFK Arena 设计 | 仙途远征修仙化改造 |
|------|---------------|-------------------|
| 离线收益 | 12h 上限，离线粉尘占日产出 ~60%，经验 ~90% | **闭关修炼**：12h 上限，离线产出灵石+修为，占日产出 60-90% |
| 战斗 | 5 人阵容，前后排站位，自动战斗+手动大招 | **斗法**：5 修士阵容，前中后排站位，自动施法+手动绝学 |
| 阵营 | 6 阵营循环克制 | **4 势力循环克制**：正道→魔道→妖族→佛门→正道（简化为 4 个） |
| 升级 | 等级+粉尘升级 | **修炼升级**：修为+灵尘升级 |
| 升品质 | 15 阶品质，吃同名卡升阶 | **突破境界**：练气→筑基→金丹→元婴→化神→...，消耗同名修士碎片突破 |
| 关卡 | 每章 40 小节，每 4 小节卡点 | **境界推进**：每境界 40 关，每 4 关一个瓶颈，瓶颈关 = 渡劫战 |
| 抽卡 | 30 抽保底紫+，心愿单 | **寻仙**：30 抽保底紫+，心愿单（定向修士） |
| 共鸣水晶 | 5 人共享等级 | **万象阵**：5 修士共享修炼等级（P1） |
| 限时礼包 | 关卡里程碑触发 | **机缘礼包**：突破境界时触发限时礼包（P1） |

## 6. 需求池

### 6.1 P0 需求（必须 — MVP 上线必备）

| 编号 | 需求 | 优先级 | 验收标准 | 估算（人天） |
|------|------|--------|----------|-------------|
| P0-01 | **闭关收益系统（离线收益）** | P0 | **Given** 玩家关闭游戏时记录时间戳，**When** 玩家重新打开游戏，**Then** 显示离线期间累积的灵石和修为（上限 12h），玩家点击领取后资源入账；前 60 分钟每 ~50 秒可领取 1 次（共 70 次）。离线收益 = 挂机产出率 × 离线时长 × 系数（灵石 0.6，修为 0.9）。 | 3 |

| P0-03 | **四势力克制系统** | P0 | **Given** 攻方修士势力克制守方，**When** 造成伤害时，**Then** 伤害 ×1.25；**Given** 攻方被克制，**When** 造成伤害，**Then** 伤害 ×0.75。同势力 5 人队伍 +25% 攻击和生命。四势力循环：正道→魔道→妖族→佛门→正道。**检查清单：** [ ] 4 势力数据定义 [ ] 克制关系查表 [ ] 同势力加成 [ ] 战斗中克制提示 | 2 |

### 7.2 核心用户流程

#### 流程 1：每日登录 → 收菜 → 推关

```
启动游戏
  → 加载存档
  → 计算离线收益
  → 主界面（显示"闭关结束"弹窗）
  → 点击领取 → 灵石/修为入账 + 动画反馈
  → 点击"境界"入口 → 进入关卡地图
  → 选择当前关卡 → 进入斗法
  → 自动战斗 + 手动绝学
  → 胜利 → 结算奖励（灵石+修为+碎片）
  → 回到关卡地图 → 继续推关 / 卡关返回升级
```

#### 流程 2：卡关 → 升级 → 突破

```
渡劫失败（卡关）
  → 返回主界面
  → 提示"修为不足，建议修炼提升"
  → 进入修士界面
  → 选择主力修士 → 消耗修为+灵尘升级
  → 若有同名碎片 → 突破境界（升品质）
  → 突破成功 → 解锁新技能 + 动画
  → 返回关卡 → 重新挑战
```

#### 流程 3：攒够灵石 → 寻仙

```
灵石充足（或获得寻仙券）
  → 进入寻仙界面
  → 选择单抽/十连
  → 抽卡动画（仙光闪烁 → 修士显现）
  → 展示结果（蓝/紫/金/红，按稀有度颜色）
  → 重复修士 → 自动转化为碎片
  → 新修士 → 解锁图鉴
  → 返回 → 进入修士界面调整阵容
```

## 8. Non-goals（明确不做什么）

MVP 阶段**明确不做**以下内容：

| # | 不做的系统/功能 | 原因 | 重新评估时机 |
|---|----------------|------|-------------|

#### 属性计算公式（GDScript）

```gdscript
# hero_data.gd
extends Resource
class_name HeroData

func get_current_stat(stat_name: String) -> float:
    var base = base_stats[stat_name]
    var growth = growth_stats[stat_name]
    var quality_mult = QualityConfig.get_multiplier(quality)
    var quality_flat = QualityConfig.get_flat_bonus(quality, stat_name)
    # 属性 = (等级基础 + 等级 × 成长) × 品质系数 + 品质固定值
    return (base + level * growth) * quality_mult + quality_flat
```

### 9.4 GDScript 开发要点（给 Java/JS 开发者的建议）

#### 概念对照表

| Java/JS 概念 | GDScript 对应 | 说明 |
|-------------|---------------|------|
| Class | `class_name` + `extends` | GDScript 每个文件默认一个类 |
| `new Object()` | `.new()` | `var hero = HeroData.new()` |
| Interface | 无直接对应 | 用基类继承或 Duck Typing |
| Event / EventEmitter | **Signal** | `signal level_up` / `level_up.emit()` |
| `addEventListener` | `signal.connect(callable)` | `hero.level_up.connect(_on_level_up)` |
| Constructor | `_ready()` | 节点进入场景树时调用 |
| `requestAnimationFrame` | `_process(delta)` | 每帧调用 |
| `setInterval` | `_physics_process(delta)` | 固定物理帧（默认 60fps） |
| `setTimeout` | `await get_tree().create_timer(n).timeout` | 延迟执行 |
| `Promise` / `async-await` | `await` + Coroutine | GDScript 原生协程 |
| `public` field | `@export` | 暴露到编辑器面板 |
| `static` method | `static func` | 静态方法 |
| Singleton | Autoload 单例 | 在 Project Settings 注册 |
| `this` | 隐式（或 `self`） | GDScript 默认隐式 this |

#### 关键注意事项

1. **动态类型 vs 静态类型**：GDScript 支持可选类型标注，**强烈建议使用类型标注**（对 Java/JS 开发者更友好，且有性能优势）：
   ```gdscript
   # 推荐（有类型标注）
   var spirit_stones: int = 0
   func add_reward(amount: int) -> void:
       spirit_stones += amount

   # 不推荐（无类型，易出错）
   var spirit_stones = 0
   func add_reward(amount):
       spirit_stones += amount
   ```

2. **Signal 通信**：Godot 推荐用 Signal 解耦，类似 JS 的事件驱动。系统间通信用 Signal，不要直接互相调用：
   ```gdscript
   # economy_manager.gd
   signal resource_changed(resource_type: String, new_amount: int)

   # 在 UI 中监听
   func _ready():
       EconomyManager.resource_changed.connect(_on_resource_changed)

   func _on_resource_changed(type: String, amount: int):
       if type == "spirit_stones":
           update_spirit_stone_label(amount)
   ```

3. **节点路径**：Godot 的节点树类似 DOM，用路径访问子节点：
   ```gdscript
   # 类似 document.querySelector()
   @onready var hp_bar = $VBoxContainer/HPBar
   # 或
   @onready var hp_bar = get_node("VBoxContainer/HPBar")
   ```

4. **资源加载**：使用 `preload`（编译时）或 `load`（运行时）：
   ```gdscript
   # 编译时加载（推荐用于固定资源）
   const HeroScene = preload("res://scenes/battle/hero_unit.tscn")

   # 运行时加载（用于动态资源）
   var sprite = load("res://assets/sprites/heroes/" + hero_id + ".png")
   ```

5. **JSON 读取**：
   ```gdscript
   func load_json(path: String) -> Dictionary:
       var file = FileAccess.open(path, FileAccess.READ)
       if not file:
           push_error("无法打开配置文件: " + path)
           return {}
       var json_text = file.get_as_text()
       file.close()
       var json = JSON.new()
       var error = json.parse(json_text)
       if error != OK:
           push_error("JSON 解析错误: " + path)
           return {}
       return json.data
   ```

### 9.5 离线收益实现方案

```
离线收益实现流程：
┌─────────────┐     ┌──────────────┐     ┌─────────────────┐
│ 游戏退出时   │────→│ 存档写入     │────→│ 记录退出时间戳   │
│ SaveManager  │     │ last_played  │     │ Time.get_       │
│ .save_game() │     │ _timestamp   │     │ unix_time()     │
└─────────────┘     └──────────────┘     └─────────────────┘

┌─────────────┐     ┌──────────────┐     ┌─────────────────┐
│ 游戏启动时   │────→│ 读取存档     │────→│ 计算离线时长     │
│ SaveManager  │     │ last_played  │     │ elapsed = now   │
│ .load_game() │     │ _timestamp   │     │  - last_played  │
└─────────────┘     └──────────────┘     └────────┬────────┘
                                                   │
                                          ┌────────▼────────┐
                                          │ elapsed > 12h?  │
                                          │  是 → cap = 12h │
                                          │  否 → cap =     │
                                          │       elapsed   │
                                          └────────┬────────┘
                                                   │
                           ┌───────────────────────▼───────────────────────┐
                           │  灵石收益 = 挂机灵石/h × cap × 0.6              │
                           │  修为收益 = 挂机修为/h × cap × 0.9              │
                           └───────────────────────┬───────────────────────┘
                                                   │
                                          ┌────────▼────────┐
                                          │ 弹出闭关收益弹窗  │
                                          │ 显示数值+动画    │
                                          │ 玩家点击领取     │
                                          └─────────────────┘
```

**GDScript 实现：**

```gdscript
# idle_system.gd
extends Node

const MAX_OFFLINE_HOURS: float = 12.0

func calculate_offline_rewards(last_timestamp: int) -> Dictionary:
    var now = Time.get_unix_time_from_system()
    var elapsed_hours = (now - last_timestamp) / 3600.0

    # 上限 12 小时
    elapsed_hours = min(elapsed_hours, MAX_OFFLINE_HOURS)

    # 不足 1 分钟不计算
    if elapsed_hours < 1.0 / 60.0:
        return {}

    var balance = load_json("res://data/balance.json").idle
    var spirit_stones = int(balance.spirit_stone_rate_per_hour * elapsed_hours * balance.offline_coefficient.spirit_stones)
    var exp = int(balance.exp_rate_per_hour * elapsed_hours * balance.offline_coefficient.exp)

    return {
        "spirit_stones": spirit_stones,
        "exp": exp,
        "offline_hours": elapsed_hours
    }
```

### 9.6 移动端适配注意事项

| 问题 | 方案 | 具体操作 |
|------|------|----------|
| **屏幕适配** | 使用 Godot 的 CanvasItem 缩放 + anchors | 设计基准分辨率 1080×1920（竖屏），使用 `stretch_mode = canvas_items`，`aspect = keep` |
| **安全区域** | 避免刘海/圆角遮挡 UI | 使用 `DisplayServer.get_display_safe_area()` 获取安全区域，顶部/底部留 padding |
| **触控输入** | 所有交互适配触摸 | 使用 `Control` 节点的 `gui_input` 信号，而非鼠标事件；按钮尺寸 ≥ 48×48 dp |
| **内存管理** | 移动端内存有限 | 修士立绘按需加载（`load` 而非 `preload`），切换场景时 `queue_free()` 释放；纹理压缩用 ETC2（Android）/ ASTC（iOS） |
| **性能优化** | 避免每帧创建对象 | 战斗中的伤害数字用对象池（Object Pool），不 `instantiate` / `queue_free` |
| **包体控制** | 目标 ≤ 50MB | 音频用 OGG（BGM）/ WAV（短 SFX）；纹理用 WebP；按需下载扩展资源（后续） |
| **后台/前台切换** | 正确处理离线收益 | 监听 `Notification.will_focus_out` 信号，退出时存档；前台时计算离线收益 |
| **发热/耗电** | 放置游戏常驻后台 | `_process` 中非必要逻辑降频（如每 0.5s 更新一次 UI 而非每帧）；战斗外关闭物理引擎 |

**项目设置建议（project.godot）：**

```ini

window/size/viewport_height = 1920
window/stretch/mode = "canvas_items"
window/stretch/aspect = "keep"

[rendering]
renderer/rendering_method = "mobile"  # 移动端用 mobile 渲染器
textures/vram_compression/import_etc2_astc = true

[input_devices]
pointing/emulate_touch_from_mouse = true  # 编辑器内测试触控
```

### 9.7 商业化技术准备

#### IAP（应用内购买）

| 平台 | 方案 | 说明 |
|------|------|------|
| Android | **Godot IAP Plugin** 或 **Google Play Billing** | Godot 4 有社区维护的 `godot-google-play-billing` 插件 |
| iOS | **Godot IAP Plugin** 或 StoreKit | Godot 4 有 `godot-ios-plugins` 中的 InAppStore |
| 跨平台 | **Deferred 自研封装层** | 建议封装统一接口 `IAPManager.purchase(product_id)`，内部按平台分发 |

**建议架构：**

```gdscript
# iap_manager.gd (Autoload 单例)
extends Node
signal purchase_success(product_id: String)
signal purchase_failed(product_id: String, error: String)

# 平台检测
var _platform: String = ""
var _payment: Node = null

func _ready():
    _platform = OS.get_name()
    if _platform == "Android":
        _payment = Engine.get_singleton("GodotPayment")
    elif _platform == "iOS":
        _payment = Engine.get_singleton("InAppStore")

func purchase(product_id: String) -> void:
    if _platform == "Android":
        _payment.purchase(product_id)
    elif _platform == "iOS":
        _payment.purchase({ "product_id": product_id })
    # 监听回调...

func restore_purchases() -> void:
    # 恢复已购商品（iOS 必须）
    pass
```

**MVP 阶段 IAP 准备：**
1. 先做好 `IAPManager` 接口封装，但**不接真实 SDK**
2. 用本地 Mock 实现（购买直接加仙玉），验证付费流程
3. 上线前接入真实 SDK

#### 广告 SDK（MVP 不做，预留）

| 广告类型 | 方案 | 时机 |
|----------|------|------|
| 激励视频 | AdMob / 穿山甲 | P2 阶段，看广告获额外闭关收益 |
| 插屏广告 | 不推荐 | 放置游戏体验敏感，不建议 |

**技术预留：** 封装 `AdManager.show_rewarded_ad()` 接口，MVP 阶段 Mock 实现。

#### 服务器端（MVP 最小化）

| 功能 | MVP 方案 | 后续方案 |
|------|---------|----------|
| 存档 | 本地存档（文件） | P1 加云端同步 |
